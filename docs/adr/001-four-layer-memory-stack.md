# ADR-001: 四层记忆栈（L0-L3）设计

**状态**：已采纳
**日期**：2026-04-08
**决策者**：MemPalace 架构团队

---

## 背景

传统 AI 助手面临"上下文窗口"困境：
- **问题 1**：粘贴所有历史记录到上下文 → 19.5M tokens，成本高昂且响应缓慢
- **问题 2**：不保留历史记录 → AI 无法记住关键事实、用户偏好、项目细节
- **问题 3**：简单摘要 → 丢失细节，无法进行深度搜索

**需求**：
- 低唤醒成本（<200 tokens）
- 高召回率（关键信息立即可用）
- 按需深度搜索（不限制查询深度）
- 完全本地（无 API 调用）

---

## 决策

采用**四层记忆栈**架构，按信息重要性和访问频率分层加载：

| 层级 | 内容 | Token 成本 | 加载时机 |
|------|------|-----------|----------|
| **L0 - Identity** | AI 身份定义（~50 tokens） | 始终加载 | 每次会话 |
| **L1 - Critical Facts** | 关键事实（团队、项目、偏好） | ~120 tokens (AAAK) | 始终加载 |
| **L2 - Room Recall** | 特定主题的近期记忆 | 按需 (~200-500) | 话题相关时 |
| **L3 - Deep Search** | 全宫殿语义搜索 | 按需 (无限制) | 主动查询时 |

**核心原则**：
1. **分层加载**：只加载当前需要的信息
2. **压缩存储**：L1 使用 AAAK 压缩（30x 缩小）
3. **语义检索**：L3 使用 ChromaDB 向量搜索
4. **本地优先**：所有数据存储在用户机器

---

## 后果

### 正面影响

**性能提升**：
- 唤醒成本从 19.5M tokens 降至 ~170 tokens（L0 + L1）
- 响应时间显著减少（无需加载整个历史）
- 可扩展性增强（宫殿大小不影响唤醒成本）

**质量提升**：
- 关键信息始终可用（L0 + L1）
- 深度搜索不限制（L3）
- 保留原始内容（非摘要），确保准确性

**成本降低**：
- 零 API 调用（完全本地）
- 减少 token 使用（~170 vs 19.5M）
- 无云服务费用

### 负面影响

**复杂度增加**：
- 需要管理四层逻辑
- AAAK 压缩需要额外处理
- 需要智能的加载策略

**维护成本**：
- 需要维护压缩/解压缩逻辑
- 需要优化 L1 内容（哪些事实是"关键"的）
- 需要管理索引和向量存储

### 风险

- **L1 过时**：关键事实可能随时间变化，需要更新机制
- **L2 召回偏差**：特定主题的召回可能遗漏相关记忆
- **L3 搜索延迟**：大规模宫殿的向量搜索可能较慢

### 缓解措施

- **自动更新**：Hook 自动保存机制确保 L1 持续更新
- **多路召回**：L2 结合时间、主题、语义多维召回
- **索引优化**：ChromaDB 持久化和索引优化

---

## 替代方案

### 方案 A：单层扁平存储

**描述**：所有记忆存储在同一层级，每次会话加载全部

**优点**：简单，无需分层逻辑

**缺点**：
- 唤醒成本高（19.5M tokens）
- 响应缓慢
- 无法扩展

**拒绝原因**：不满足低唤醒成本需求

### 方案 B：两层架构（热/冷）

**描述**：只有热数据（最近）和冷数据（历史）

**优点**：比单层更简单

**缺点**：
- 热数据定义模糊
- 无法区分"关键"和"最近"
- 缺乏身份层

**拒绝原因**：无法满足关键信息始终可用的需求

### 方案 C：纯云端方案

**描述**：使用云服务（如 GPT-4 的记忆功能）

**优点**：无本地维护成本

**缺点**：
- 隐私风险
- 依赖网络
- 持续 API 成本
- 无法控制存储

**拒绝原因**：不满足完全本地需求
